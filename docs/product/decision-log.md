# 제품 의사결정 기록

## 2026-07-15 — 모바일 웹 우선

- 결정: P0는 모바일 웹으로 만들고 방문자 응답은 앱 출시 후에도 웹에서 가능하게 유지한다.
- 이유: Instagram·KakaoTalk 링크에서 설치나 로그인 없이 바로 응답해야 바이럴 퍼널이 유지된다.
- 결과: 네이티브 앱은 프로필 주인의 알림·관리 재방문이 검증된 뒤 별도 검토한다.

## 2026-07-15 — 주인 10장, 방문자 3장

- 결정: 팩 주인은 10장 전부에 답하고 방문자는 공통 핵심 1장과 균형 추출 2장만 필수로 답한다.
- 이유: 주인은 자기 프로필을 만들 동기가 있지만 방문자에게 10장을 요구하면 링크 전환이 떨어진다.
- 결과: 3장 완료 후 즉시 비교하며, 2장 추가 응답은 선택 사항으로 둔다.

## 2026-07-15 — 관계는 방문자가 직접 선택

- 결정: 팩 주인이 공유 전에 관계를 지정하지 않는다.
- 이유: 공개 링크는 친구뿐 아니라 온라인 친구와 SNS 팔로워도 참여할 수 있다.
- 결과: 방문자가 관계와 알게 된 시점을 직접 선택하고 관계별 프로필 집계에 사용한다.

## 2026-07-15 — 방문자를 다음 주인으로 전환

- 결정: 비교 결과의 Primary CTA는 `나도 이 팩으로 시작하기`다.
- 이유: 방문자는 단순 응답자가 아니라 다음 링크를 만드는 잠재적 프로필 주인이다.
- 결과: CTA를 누르면 탐색 화면 없이 동일한 팩의 셀프 10장 응답으로 이동한다.

## 2026-07-15 — 사용자 팩 제작은 P1

- 결정: P0는 공식 팩으로 루프를 검증하고 P1에 선택·조합형 팩 메이커를 추가한다.
- 이유: 질문 공급은 확장해야 하지만 첫 프로토타입에 공개 마켓까지 포함하면 핵심 가설 검증이 느려진다.
- 결과: P1 사용자는 관계·주제·분위기를 고르고 추천 15장 중 10장을 선택해 링크 공개 팩을 만든다.

## 2026-07-15 — P0 공유 대상은 특정 팩 링크

- 결정: P0에서 주인이 외부에 공유하는 기본 대상은 공개 프로필이 아니라 자신이 완료한 특정 팩 플레이 링크다.
- 이유: 미참여 방문자에게 셀프 응답을 숨긴 채 3장 응답으로 바로 진입해야 비교 보상과 바이럴 퍼널이 유지된다.
- 결과: P0 프로필은 주인용 누적 화면으로 두고, 공개 프로필 링크와 프로필에서 참여할 대표 팩을 고르는 기능은 P1에서 추가한다.

## 2026-07-15 — 제품 콘텐츠와 프로필은 AI 없이 구성

- 결정: 현재 P0·P1 범위에서 AI로 질문, 방문자용 문구, 제목, 커버, 프로필 소개문이나 성격 요약을 생성하지 않는다. 프로필은 원본 A/B 질문, 셀프 선택, 관계별 좌우 선택 수로 구성한다.
- 이유: 모든 제품 표현이 사람이 검수한 콘텐츠와 실제 답변으로 추적 가능해야 하고, 근거 없는 성격 추론이나 고정 유형을 만들지 않아야 한다.
- 결과: P0 공식 팩의 주인용·방문자용 질문은 모두 사람이 작성·검수한다. P1 팩 메이커는 검증된 질문 은행, 기본 제목, 커버 템플릿을 사용한다. 유효하게 완료된 친구 응답 한 건을 `시선` 한 건으로 세고, 카드 응답은 팩 플레이·카드 버전·관계별로 집계한다. 관계 레이어는 같은 관계 완료 방문자 3명 이상, 질문별 선택 수는 같은 관계·질문 카드 응답 3개 이상에서만 공개한다.

## 2026-07-15 — P0 첫 공식 팩은 오래된 친구팩 하나로 시작

- 결정: P0 첫 테스트는 `오래된 친구팩` 1개만 제공한다.
- 이유: 여러 관계와 민감도를 동시에 검수하기보다 공유 장벽이 낮고 관계 맥락이 분명한 하나의 팩으로 핵심 루프를 먼저 검증한다.
- 결과: 스타터·첫인상·직장동료·썸·연애·솔직한 나 팩은 오래된 친구팩의 핵심 지표를 확인한 뒤 순차적으로 검토한다.

## 2026-07-15 — 무가입 방문자 철회는 비밀 관리 링크로 제공

- 결정: 방문자 응답 제출 직후 추측하기 어려운 비밀 관리 링크를 발급해 현재 브라우저에 저장하고 복사할 수 있게 한다. 방문자는 계정이나 이름 없이 이 링크로 응답을 철회한다.
- 이유: 익명 참여를 유지하면서도 본인의 응답만 철회할 수 있는 최소 소유권 증명이 필요하다.
- 결과: 링크 분실 시 신원 확인이나 재발급은 제공하지 않는다. 철회 시 응답과 집계를 제거하고 관리 링크를 폐기하며, 완료된 1:1 링크는 다시 열지 않는다.

## 2026-07-15 — P0 계정 연결과 알림은 이메일로 통일

- 결정: 팩 주인의 P0 계정 연결은 이메일 매직 링크만 사용하고, 응답 도착 알림도 같은 계정 이메일로만 보낸다.
- 이유: 소셜 로그인과 웹 푸시를 함께 지원하지 않아도 하나의 검증된 연락 수단으로 저장·재방문 루프를 완성할 수 있다.
- 결과: P0에서 비밀번호, 소셜 로그인, 웹 푸시, 방문자 재촉 알림은 구현하지 않는다.

## 2026-07-15 — P0 공유 문구에는 표시 이름을 사용하지 않음

- 결정: P0에서는 주인의 표시 이름이나 닉네임을 입력받지 않고, 초대·질문·비교 화면에 `친구`, `이 사람` 같은 검수된 일반 문구를 사용한다.
- 이유: 공개 프로필이 없는 P0에서 추가 개인정보와 이름 검수·신고 흐름을 만들지 않고도 핵심 비교 루프를 완성할 수 있다.
- 결과: 표시 이름과 프로필 이미지는 기존 계획대로 P1에서 공개 범위와 신고 규칙을 함께 정한 뒤 도입한다.

## 2026-07-15 — P0 draft 귀속은 같은 브라우저에서만 지원

- 결정: 셀프 답변을 시작한 브라우저에서 이메일 매직 링크를 열었을 때만 로그인과 익명 draft 귀속을 진행한다.
- 이유: 다른 기기로 비밀 draft 권한을 안전하게 전달하는 별도 handoff 흐름 없이 P0 저장·복구 루프를 완성한다.
- 결과: 다른 기기에서 링크를 열면 로그인을 진행하지 않고 원래 브라우저로 돌아가도록 안내한다. 복수 draft와 cross-device handoff는 실제 이탈을 확인한 뒤 검토한다.
