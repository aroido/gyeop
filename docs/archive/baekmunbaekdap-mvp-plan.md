# 백문백답 소셜 웹서비스 MVP 개발 계획

작성일: 2026-07-14  
상태: 제품 가설 검증용 초안  
기본 가정: 1~2명의 소규모 팀이 2~4주 안에 모바일 웹 MVP를 출시한다.

## 1. 한 줄 제품 정의

사용자가 짧은 문답으로 `나 사용설명서`를 만들고, 친구가 답을 예상한 뒤 서로의 일치율을 확인하며 자기 문답을 다시 만드는 소셜 자기소개 게임.

## 2. 이번 개발의 목표

완성형 SNS나 100문항 데이터베이스를 만드는 것이 아니다. 아래 바이럴 루프가 실제 사용자 사이에서 반복되는지만 검증한다.

`10문항 답변 → 결과 카드 생성 → 친구에게 공유 → 친구가 3문항 예상 → 결과 공개 → 친구도 자기 문답 생성`

첫 출시에서 검증할 핵심 질문은 세 가지다.

1. 사용자가 10문항을 끝까지 작성하는가?
2. 완성된 결과를 실제 친구에게 공유하는가?
3. 링크로 들어온 친구가 구경만 하지 않고 자기 문답까지 만드는가?

## 3. 제품 원칙

- 가입보다 체험이 먼저다. 첫 문항 시작에 이메일·전화번호·소셜 로그인을 요구하지 않는다.
- 첫 관계 경험은 무료다. 친구 답변을 보는 순간까지 광고나 결제로 막지 않는다.
- 100문항은 목표가 아니라 누적 결과다. 첫 방문에는 10문항만 제공한다.
- 공유되는 결과가 제품의 얼굴이다. 문답 화면보다 결과 카드와 링크 미리보기 품질을 우선한다.
- 공개 범위는 작성자가 결정한다. 답변은 기본적으로 친구 링크 안에서만 보이고, 항목별 숨김과 전체 삭제를 제공한다.
- 어머니 성함, 첫 학교, 반려동물 이름처럼 계정 복구 질문에 쓰일 수 있는 개인정보는 질문으로 수집하지 않는다.

## 4. MVP 사용자 흐름

### 작성자 흐름

1. 랜딩에서 예시 결과 카드와 `내 사용설명서 만들기` 버튼을 본다.
2. 닉네임만 입력하고 10문항에 답한다.
3. 선택형 답변을 기준으로 3개의 성향 태그와 한 줄 소개가 만들어진다.
4. 공개할 답변을 확인하고 결과를 발행한다.
5. 카카오톡·모바일 공유 시트·링크 복사로 친구에게 보낸다.
6. 친구가 예상 게임을 마치면 작성자는 활동 알림 화면에서 결과를 확인한다.

### 친구 흐름

1. 공유 링크에서 작성자의 결과 카드와 공개 답변 일부를 본다.
2. 실제 답을 보기 전에 무작위 3문항의 답을 예상한다.
3. 적중 개수와 두 사람의 공통점·차이점을 확인한다.
4. `이번엔 내가 문제 내기`를 눌러 자기 문답을 만든다.

### 재방문 흐름

- 작성자는 새 친구 반응을 확인하고 추가 질문팩을 작성한다.
- 10개씩 답할 때마다 프로필이 누적되어 최종적으로 백문백답이 완성된다.

## 5. 첫 질문팩 구성

첫 팩은 결과 계산과 친구 예상 게임을 위해 구조화한다.

- 8개: 2~4지선다형 취향·관계 질문
- 2개: 짧은 자유응답
- 예상 게임에는 선택형 문항만 사용
- 민감 질문은 넣지 않고 `음식`, `연락`, `주말`, `여행`, `갈등`, `소비`, `연애관`, `친구관계`처럼 대화가 이어지는 주제를 사용

AI 요약은 첫 버전에 넣지 않는다. 선택지별 태그 점수를 합산하는 결정론적 규칙으로 결과를 만들면 비용이 없고 결과 재현성과 테스트가 쉽다.

## 6. 기능 범위

### 반드시 포함

- 모바일 우선 반응형 웹
- 닉네임 기반 익명 시작
- 10문항 작성, 임시 저장, 이어쓰기
- 결과 태그와 한 줄 소개 생성
- 답변별 공개·비공개 설정
- 공개 링크와 추측 게임
- 동적 Open Graph 이미지 및 저장 가능한 결과 카드
- 공유 시트, 링크 복사, 카카오 공유 진입점
- 작성자용 반응 목록
- 본인 콘텐츠 전체 삭제
- 신고, 기본 금칙어, 요청 제한
- 퍼널 이벤트 기록

### 첫 출시에서 제외

- 광고와 결제
- 100문항 일괄 작성
- 네이티브 iOS·Android 앱
- 팔로우, 친구 목록, 피드, 댓글, 채팅
- 익명 자유질문
- AI 성격 분석
- MBTI 등 심리검사 정확성 주장
- 커플 궁합, 브랜드 질문팩, 프로필 꾸미기

## 7. 권장 기술 구조

### 권장안: 관리형 단일 웹 애플리케이션

- 웹: Next.js App Router + TypeScript
- 데이터·인증: Supabase Auth + PostgreSQL + Row Level Security
- 배포: Vercel 또는 동등한 관리형 호스팅
- 스타일: 프로젝트가 선택한 단일 CSS 방식만 사용
- 오류 추적: Sentry 또는 동등한 서비스
- 제품 분석: 초기에는 핵심 이벤트를 자체 `events` 테이블에 저장하고, 트래픽 증가 후 전문 분석 도구로 이전

선정 이유는 하나의 TypeScript 저장소에서 화면, 서버 로직, 동적 공유 이미지까지 처리할 수 있고 운영 인력이 적어도 되기 때문이다. Next.js는 경로별 동적 Open Graph 이미지 생성을 공식 지원한다. Supabase는 개인정보 입력 없는 익명 사용자와 이후 영구 계정 연결을 지원하지만, 익명 계정 남용 방지를 위한 CAPTCHA·요청 제한과 RLS 검토가 필요하다.

참고 문서:

- [Next.js 동적 Open Graph 이미지](https://nextjs.org/docs/app/api-reference/file-conventions/metadata/opengraph-image)
- [Supabase 익명 로그인](https://supabase.com/docs/guides/auth/auth-anonymous)
- [Supabase Auth와 RLS](https://supabase.com/docs/guides/auth)
- [Web Share API와 브라우저 지원 제약](https://developer.mozilla.org/en-US/docs/Web/API/Web_Share_API)

Web Share API는 모든 브라우저에서 동일하게 지원되지 않으므로 `공유 시트 → 링크 복사 → 이미지 저장` 순서의 폴백을 반드시 둔다.

## 9. 최소 데이터 모델

### `profiles`

- `id`
- `owner_id`: 익명 또는 영구 사용자 ID
- `slug`: 공유용 추측 불가능 식별자
- `nickname`
- `status`: draft/published/deleted
- `result_code`
- `created_at`, `published_at`

### `questions`

- `id`, `pack_id`
- `prompt`
- `type`: choice/short_text
- `options`
- `sensitivity_level`
- `sort_order`
- `is_active`

### `answers`

- `profile_id`, `question_id`
- `choice_key` 또는 `text_value`
- `visibility`: public/after_guess/private

### `guess_sessions`와 `guesses`

- 대상 프로필
- 익명 방문자 ID
- 출제된 질문 ID와 예상 답
- 정답 수
- 완료 시각

한 방문자가 반복 제출해 반응 수를 조작하지 못하도록 대상 프로필과 방문자 기준으로 중복을 제한한다.

### `events`

- 익명 세션 ID
- 이벤트 이름
- 유입 공유 ID
- 필요한 최소 속성
- 발생 시각

자유응답 원문은 분석 이벤트에 복제하지 않는다.

### 제안하는 초기 통과 기준

아래 숫자는 업계 평균이 아니라 MVP의 내부 go/no-go 기준이다.

- 문답 시작자 중 10문항 완료율 60% 이상
- 완료까지 걸린 시간 중앙값 3분 이하
- 프로필 발행자 중 공유 행동률 25% 이상
- 공유 링크 방문자 중 예상 게임 완료율 35% 이상
- 예상 게임 완료자 중 자기 문답 시작률 20% 이상
- 작성자 1명당 새 프로필 발행 수 0.5 이상이면 반복 개선, 1.0 이상이면 자생 바이럴 가능성 확인
- 삭제·신고 사유에서 의도하지 않은 공개나 불쾌한 질문이 반복되면 성장 실험보다 안전 개선을 우선

## 11. 단계별 개발 일정

### 0단계 — 제품 명세와 질문팩 확정, 1~2일

- 첫 10문항과 선택지, 태그 점수표를 작성한다.
- 공개 범위와 삭제 정책을 결정한다.
- 위 이벤트 정의와 지표 계산식을 명세한다.

완료 조건: 같은 입력에 항상 같은 결과가 나오는 점수표와 화면별 와이어 흐름이 문서화되어 있다.

### 1단계 — 클릭 가능한 가짜 흐름과 수요 확인, 2~3일

- 랜딩, 10문항, 결과 카드, 친구 예상 결과만 더미 데이터로 연결한다.
- 실제 사용자 10명에게 모바일로 수행시킨다.
- 이해되지 않는 문구, 중도 이탈 문항, 공유하고 싶지 않은 결과 표현을 수정한다.

완료 조건: 10명 중 8명 이상이 설명 없이 흐름을 끝내고, 5명 이상이 실제 친구에게 보낼 의향을 표현한다.

### 2단계 — 기능형 MVP, 7~10일

- 익명 세션, 데이터 모델, RLS와 마이그레이션을 구현한다.
- 작성·발행·공개·예상·결과·삭제 흐름을 연결한다.
- 동적 OG 이미지와 공유 폴백을 구현한다.
- 이벤트와 오류 추적을 연결한다.
- 요청 제한, CAPTCHA, 신고를 적용한다.

완료 조건: 새 브라우저 두 개를 사용한 전체 작성자→친구→새 작성자 흐름이 자동 E2E 테스트와 실제 모바일 기기에서 통과한다.

### 3단계 — 비공개 베타, 1주

- 서로 모르는 최소 3개 친구 집단에서 총 50명의 시드 사용자를 모집한다.
- 기능 추가 없이 퍼널과 인터뷰 데이터를 수집한다.
- 가장 큰 이탈 지점 한 곳만 수정한다.

완료 조건: 이벤트 누락률 1% 미만이며 각 퍼널 지표를 유입 집단별로 계산할 수 있다.

### 4단계 — 공개 베타 판단

- 200개 이상의 발행 프로필 또는 2주간의 데이터를 기준으로 판단한다.
- 바이럴 지표가 부족하면 결제를 붙이지 않고 질문·결과 카드·친구 게임을 개선한다.
- 지표를 통과하면 추가 질문팩과 작성자 재방문 알림을 먼저 개발한다.
- 반복 사용이 확인된 다음에만 광고·유료 질문팩 실험을 시작한다.

## 12. 테스트와 검증

### 단위 테스트

- 점수 계산은 같은 답변에 같은 결과를 반환한다.
- 동점 처리 결과가 고정되어 있다.
- 비공개 및 예상 후 공개 답변이 각 상태에서 올바르게 필터링된다.
- 삭제된 프로필은 공개 경로와 OG 이미지에서 조회되지 않는다.

### 통합 테스트

- 익명 사용자가 자기 프로필만 수정·삭제할 수 있다.
- 방문자는 공개 데이터만 읽을 수 있다.
- 추측 제출 전에는 `after_guess` 답변을 받을 수 없다.
- 반복 추측과 과도한 프로필 생성을 제한한다.

### E2E 테스트

- 작성 → 발행 → 공유 링크 방문 → 3문항 예상 → 결과 → 새 프로필 생성
- 작성 중 새로고침과 재접속
- 공유 API 미지원 브라우저의 링크 복사 폴백
- 신고와 전체 삭제
- iOS Safari, Android Chrome, 카카오톡 인앱 브라우저의 핵심 흐름

### 운영 검증

- 공개 전 백업과 복구 절차를 한 번 실행한다.
- 이벤트 퍼널이 실제 DB 수와 일치하는지 표본 대조한다.
- 에러율, API 지연, 발행 실패, 신고율을 확인할 최소 운영 화면을 둔다.
- 저장소의 정식 완료 전에는 `./scripts/ai-verify --mode full` 또는 저장소가 제공하는 `./scripts/run-ai-verify`를 통과해야 한다.

## 13. 주요 위험과 대응

### 10문항도 길게 느껴짐

- 첫 화면에서 예상 소요 시간을 보여주고 한 화면 한 문항으로 구성한다.
- 3문항 완료 시 임시 결과를 보여주되 최종 카드를 위해 계속하도록 유도한다.

### 친구가 링크만 보고 이탈함

- 프로필 전체보다 `내 답을 먼저 맞혀봐`를 첫 CTA로 둔다.
- 가입 없이 3문항 예상까지 가능하게 한다.

### 공개 답변 때문에 관계 문제가 생김

- 발행 직전 실제 공개 화면 미리보기를 제공한다.
- 문항별 공개 범위와 즉시 삭제를 제공한다.
- 답변을 제3자가 재공유할 수 있다는 안내를 명확히 한다.

### 익명 계정과 링크가 남용됨

- CAPTCHA, IP·계정 기준 요청 제한, 신고, 금칙어를 적용한다.
- 익명 사용자의 공개 발행 수를 제한하고 오래된 미발행 익명 계정을 정리한다.

### 결제를 너무 빨리 붙여 바이럴이 끊김

- 공유→친구 완료→신규 작성의 기본 루프는 무료로 유지한다.
- 유료화는 추가 질문팩, 꾸미기, 상세 관계 리포트처럼 기본 공개권과 분리한다.

## 14. 첫 출시 뒤 우선순위

지표가 확인되면 아래 순서로 확장한다.

1. 친구 반응 알림과 추가 10문항 팩
2. 커플·절친·새학기 등 상황별 질문팩
3. 결과 카드 테마와 미니홈피식 꾸미기
4. 계정 전환과 여러 기기 복구
5. 보상형 광고 또는 유료 질문팩 A/B 테스트
6. 브랜드 협업용 질문팩 제작 도구

피드와 팔로우는 이 단계에서도 보류한다. 링크 기반 루프만으로 재방문이 만들어지지 않을 때 피드를 추가해도 문제를 해결하지 못할 가능성이 크다.

## 15. 최종 수용 조건

MVP는 다음을 모두 만족할 때 개발 완료로 간주한다.

- 신규 사용자가 가입 없이 10문항을 작성하고 결과를 발행할 수 있다.
- 공개 링크에 작성자가 허용한 데이터만 노출된다.
- 친구가 가입 없이 3문항을 예상하고 결과를 확인할 수 있다.
- 친구가 동일 흐름에서 자기 문답 생성을 시작할 수 있다.
- 결과 링크에 프로필별 이미지와 제목이 표시된다.
- 공유 기능 미지원 환경에서도 링크 복사가 작동한다.
- 작성자가 자기 프로필과 답변을 즉시 삭제할 수 있다.
- 핵심 퍼널 12개 이벤트가 중복 기준과 함께 기록된다.
- 권한, 공개 범위, 점수 계산, 전체 바이럴 흐름 테스트가 통과한다.
- 저장소 전체 검증 명령이 성공한다.

## 16. 지금 확정해야 할 첫 번째 제품 결정

개발 착수 전 가장 먼저 정할 것은 기술 스택이 아니라 **첫 질문팩의 대상 관계**다. 범용 백문백답보다 `절친이 나를 얼마나 아는지` 한 가지 상황으로 시작하는 편이 문항, 문구, 공유 이유와 성과 측정을 선명하게 만든다. 첫 베타에서는 이 한 가지 사용 사례만 구현하는 것을 권장한다.
