# Issue 54 구현 스펙: [Frontend] 첫인상·직장동료·솔직한 나팩 로컬 플레이 구현

Status: Reviewed
Issue: https://github.com/aroido/gyeop/issues/54

## 목표

개발 환경 메인에서 첫인상팩·직장동료팩·솔직한 나팩을 선택해 각 10장 A/B 질문을 끝까지 플레이하고 팩마다 독립적으로 초안을 복구할 수 있게 한다.

## 범위

- 검수된 팩 정의를 한 파일에 모으고 기존 오래된 친구팩을 포함한 네 팩이 같은 플레이 컴포넌트를 사용하게 한다.
- 새 팩 3개에 stable card id, 주인용·방문자용 질문 10장, 좌우 선택지, Signature 1장, 관계·분위기·민감도·공유 권장값을 정의한다.
- 개발 환경 메인에서 네 팩 모두 `팩 열어보기` 링크를 제공한다.
- `/play/[slug]`에서 등록된 네 팩만 열고 잘못된 slug는 404로 처리한다.
- 기존 개봉, 선택, 이전 질문, 새로고침 복구, 완료 요약, 재시작 UI를 그대로 재사용한다.
- 팩별 localStorage key를 분리하고 기존 오래된 친구팩 key `gyeop:old-friend-play:v1`은 유지한다.
- 메인과 팩 플레이 E2E를 새 팩 흐름에 맞게 보강한다.

## 제외 범위

- Supabase schema·seed·팩 발행·공유 링크·방문자 응답·프로필 집계
- 운영 환경에서 새 팩 또는 오래된 친구팩 로컬 플레이 활성화
- 새 디자인 시스템, 개봉 연출 변경, 스와이프 제스처, 새 의존성
- 사용자 팩 메이커와 AI 질문·결과 생성
- 활성 제품 SSOT의 `P0 첫 공식 팩 1개` 결정 변경

## SSOT

- `docs/product/core-feature-priority.md` 5.2, 5.3, 14: P0 첫 공식 팩은 오래된 친구팩 1개이며 나머지는 순차 검토한다.
- `docs/product/question-pack-spec.md` 2~4, 11: 팩당 10장, Signature 1장, 주인·방문자 문구, 균형 잡힌 A/B 선택지와 민감도·공유 기본값을 요구한다.
- `docs/product/decision-log.md` 2026-07-15 결정: P0 첫 테스트에서 새 팩을 정식 제공하지 않는다.

## 사용자 흐름 영향

- 개발 환경 사용자는 메인 질문팩 레일에서 네 팩 중 하나를 골라 같은 10장 셀프 응답 흐름으로 진입한다.
- 새 팩의 답변은 다른 팩 초안을 덮어쓰지 않으며, 메인 방문도 기존 초안을 수정하지 않는다.
- 방문자 응답과 새 주인 전환 흐름은 아직 구현하지 않는다.
- 운영 환경 사용자는 기존과 동일하게 모든 팩을 `준비 중`으로 보고 플레이 라우트에 접근할 수 없다.

## 디자인 영향

- `app/(public)/page.tsx`: 기존 네 카드의 색상·순서·레이아웃은 유지하고 개발 환경에서만 각 카드에 CTA를 넣는다.
- `app/play/[slug]/*`: 기존 오래된 친구팩 UI와 CSS를 그대로 공용화하고 팩 제목·질문·선택지만 데이터로 바꾼다.
- 320px에서 가로 overflow 없이 선택 버튼 44px 이상, 질문 focus 이동, reduced motion 동작을 유지한다.

## API와 데이터 영향

- API, schema, migration, auth 변경 없음.
- 브라우저 저장 key:
  - 오래된 친구팩: `gyeop:old-friend-play:v1`
  - 첫인상팩: `gyeop:first-impression-play:v1`
  - 직장동료팩: `gyeop:coworker-play:v1`
  - 솔직한 나팩: `gyeop:honest-self-play:v1`
- draft 형식 `{ version: 1, currentIndex, answers }`은 유지하고 현재 팩의 card id만 허용한다.
- 개발 전용 in-memory fixture라 카드별 topic·관계·민감도 tag와 발행 active/version은 이번 범위에 저장하지 않는다. 팩 단위 metadata와 stable card id만 두고, production seed·발행 이슈에서 reusable schema 전체를 적용한다.

## 구현 계획

1. `app/play/packs.ts`에 네 팩의 metadata, storage key, 카드 데이터를 정의하고 등록된 slug 조회 함수를 둔다.
2. 기존 `app/play/old-friend/play.tsx`와 CSS를 `app/play/[slug]/`로 옮겨 `PackPlay`가 선택된 팩 데이터를 받게 한다.
3. `app/play/[slug]/page.tsx`에서 개발 환경과 등록 slug를 확인한 뒤 공용 플레이를 렌더링한다.
4. `app/(public)/page.tsx`의 네 팩 카드를 데이터 기반으로 렌더링하되 기존 클래스와 레일 구조를 유지한다.
5. `tests/unit/packs.test.mjs`에서 네 팩의 10장, Signature 1장, unique/non-empty id, 주인용·A/B 문구, unique storage key와 새 팩 3개의 방문자용 문구를 직접 검증한다.
6. `package.json`의 `pnpm test`가 기존 환경 검증과 새 팩 data-level test를 함께 실행하게 한다.
7. 기존 E2E를 동적 route에 맞게 갱신하고 새 팩 3개의 첫 질문, 10장 완료, 저장 key 분리를 표 기반 테스트로 추가한다.

## 완료 기준

- 개발 환경 메인에 네 팩 제목, 관계·질문 수·시간·분위기·민감도·공유 권장이 보이고 네 개의 `팩 열어보기` 링크가 존재한다.
- 각 링크가 해당 `/play/<slug>`로 이동하고 팩 제목과 첫 질문을 보여 준다.
- 새 팩마다 카드가 정확히 10장이고 Signature가 정확히 1장이다.
- 모든 팩의 storage key와 팩 내부 card id가 유일하며, 모든 카드에 non-empty 주인용 질문과 겹치지 않는 A/B 선택지가 있다.
- 새 팩 3개의 모든 카드에 스펙에서 검수한 non-empty 방문자용 질문이 있다. 오래된 친구팩 방문자 문구 추가는 이번 범위가 아니다.
- 각 팩에서 이전 답 확인, reload 복구, 10장 완료, 재시작이 동작한다.
- 네 팩 초안은 서로 다른 key에 저장되며 기존 오래된 친구팩 draft와 호환된다.
- `/play/not-a-pack`은 개발 환경에서도 404다.
- production build의 모든 `/play/<slug>`는 404이고 메인에는 사용 가능한 팩 링크가 없다.
- production 메인에는 가짜 결과, 사용자 수, 사람·아바타 등 사회적 증거가 없다.
- 320x800과 430x932에서 문서 가로 overflow가 없고 주요 조작 대상은 44px 이상이다.
